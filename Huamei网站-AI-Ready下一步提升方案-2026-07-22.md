# Huamei 网站 AI-Ready 下一步提升方案

**制定日期：**2026-07-22  
**执行主线：**/Users/qu/Documents/Huamei-官网  
**目标：**不牺牲老年用户可读性和现有转化路径，建立可抓取、可理解、可引用、可追溯、可持续运营的网站内容系统。

## 一、当前基线与边界

### 本地代码已经完成

- 显式允许训练、搜索索引、实时用户代理三类主流爬虫。
- canonical、Open Graph / JSON-LD URL、站内链接和 sitemap 统一到无 .html 的主网址。
- CMS 发布、更新、改址、下架、删除的异步 IndexNow 通知代码与密钥验证端点。
- 公寓筛选参数页使用 noindex,follow，并指向基础列表页 canonical。
- sitemap 使用真实 lastmod；动态页已补 BlogPosting、发布方、栏目和关键词等数据。
- AI 引流识别同时读取 UTM 与 referrer，并记录识别方法。
- v2.1 落款已接入 7 个静态页和 1 个 CMS 动态模板，等待源头修复包后统一替换。
- 本地自动化验证 90/90 通过，npm run build 成功。

### 已做线上只读抽检

- 10 类 UA × 首页与 2 个关键页，共 30 个请求均返回 200，正文大小一致，未见挑战页。
- 抽样 TTFB：首页约 30ms，车辆页约 84ms。
- 静态资源当前约为 4 小时缓存。

### 当前边界

**这些代码尚未发布到正式站。**线上 robots.txt 仍是旧版，线上 canonical/sitemap 也尚未获得本地修正。“本地完成”不等于“线上生效”，更不等于“外部平台已留证”。

## 二、执行原则

1. **先事实，后扩写。**官网要成为业务事实正本。
2. **人类可读内容就是 AI 内容。**不建设只给机器看的页面或文案。
3. **不静默改变已审定业务口径。**政策表述先补来源、范围和审核日；需要改变原意时交业务方审核。
4. **老年人可读性与 SEO/GEO 共用一套结构。**先给结论、短段落、清楚小标题、大字号、高对比，不堆关键词。
5. **分层验收。**源码、构建、正式域名、外部平台和 AI 结果观测分别记录。

## Phase 1：建立业务事实正本（P0）

### 1.1 Official Fact Sheet

由业务方确认：

- 对外品牌名、法定公司名及两者关系。
- 地址属于到访、邮寄，还是只接受预约。
- 电话、微信、邮箱、预约时间和服务区域。
- 实际提供服务的语言。
- 各服务所需资质/执照，或“不适用特定许可证”的正式判断。
- 住房、车辆、健康等服务的包含与不包含。
- GBP、Yelp 和主要外部目录链接。

**Codex：**将结果变成可维护的事实正本，并映射到站内文案、JSON-LD、联系区和外部平台比对表。  
**验收：**核心事实零冲突；每个字段有负责人和最后审核日。

### 1.2 政策与高风险主张登记表

建立 Claim / Source / Applicability / Last reviewed / Owner / Page 登记表，优先复核：

- “比市场价低 50% 以上”。
- “先申请的人永远在前面”。
- “不会突然删除/跳过”。
- “只要资格符合就能一直住”。
- 身份例外、申请比例、办理时效或当天入住等数字化承诺。

**处理原则：**不静默改写已审核 Q&A 的核心意思。先补官方来源、适用地区/项目、例外和审核日；来源不支持时，再交业务方选择删除或收窄。  
**验收：**时效性政策主张 100% 有来源和日期；无未处理的高风险绝对表述。

## Phase 2：提升页面可引用性与老年可读性（P1）

### 2.1 关键页首屏“一段话讲清”

对首页、住房、车辆、健康、爱健康增加可见摘要，直接回答：

1. 这是什么服务。
2. 适合谁。
3. 服务范围在哪里。
4. 怎么联系或开始。
5. 不是什么、不保证什么。

保留现有品牌口号，但让 H1 或紧邻主标题承担真实主题。  
**验收：**只读每个关键页前 150–250 字，也能准确回答上述五项。

### 2.2 政策内容增加“来源与更新”模块

住房政策解读、申请攻略和时效性博客增加：

- 主要官方来源。
- 适用地区或计划。
- 首次发布日期和最后审核日。
- 作者与审核人。
- 一般信息边界，并提示以具体管理机构为准。

**验收：**可见内容与 BlogPosting、FAQPage 等结构化数据逐项一致。

### 2.3 扩展真实 FAQ

- 住房：保留已有真实 Q&A，补来源和审核日。
- 车辆/健康：从客服记录提取高频问题，先由业务方审核答案，再上页。
- 只有 FAQ 对人可见时才输出 FAQPage，不宣传为 Google 富结果保证。

### 2.4 语言策略决策

如果真实服务英语用户，建设独立 /en/... URL、可见入口、hreflang 和双语事实同步流程；如果不提供，则记录为 N/A，不制作只给机器看的英文页。

## Phase 3：代码、性能与可访问性（P1）

### 3.1 图片尺寸与布局稳定

当前待处理内容图约为：首页 4 张、车辆页 17 张、健康页 10 张、爱健康页 4 张。逐张增加正确 width/height 或可验证的 aspect-ratio，并区分真实 alt 与装饰图空 alt。

**验收：**首屏无明显图片布局跳动；尺寸、解码和懒加载策略通过源码与真实浏览器验证。

### 3.2 结构化数据和元信息

- 根据可见内容选择 Organization / LocalBusiness / Service / BlogPosting / BreadcrumbList / FAQPage。
- 用 Schema.org validator 和适用的 Google Rich Results Test 外部校验。
- 法务页 OG/Twitter 可作 P2 整齐性修复，不阻断主业务发布。

### 3.3 真实浏览器与 CWV

发布后分三层验证：

1. Lighthouse 实验室诊断。
2. PageSpeed Insights 线上诊断。
3. Search Console CWV 现场数据，有样本后作长期判断。

如果要将 4 小时缓存拉长，先为 CSS、JS 和图片引入内容指纹。

### 3.4 代理友好与表单安全

- 检查所有 CTA、导航、表单、错误提示的 label/role/state。
- 保持全键盘可用和可见焦点。
- 不收集业务不需要的医疗或健康敏感信息。
- 后端保留输入验证、频率限制和防滥用；不为 AI 代理建立绕过人类页面的通道。

## Phase 4：外部平台闭环（下一轮共同处理）

| 平台 | 动作 | 验收证据 |
|---|---|---|
| Cloudflare | 确认 Block AI Bots 关闭；核查 AI Crawl Control、WAF 和 Bot 规则顺序；保持三类爬虫可访问；配置 INDEXNOW_KEY | 后台截图、规则导出、线上 robots.txt、验证爬虫日志 |
| Google Search Console | 验证所有权，提交 sitemap，核对生成式 AI 功能参与设置与报告可用性 | 验证、sitemap 成功、设置与基线截图 |
| Bing Webmaster Tools | 验证站点，提交 sitemap，核对 IndexNow | 后台截图、一次真实发布通知和日志 |
| GA4 | 创建 ai_source、detection_method、referrer_host、landing_path 自定义维度与探索 | 实时测试事件、维度配置、基线报表 |
| GBP / Yelp / 目录 | 与 Official Fact Sheet 做规范化事实比对 | 对照表、最后核对日和负责人 |

## Phase 5：CMS 通知与持续运营（P1）

### 5.1 CMS 通知机制

发布或更新时：

    保存数据成功
      → 确定变动 URL（新 URL＋必要时旧 URL）
      → 异步 POST 到 IndexNow
      → 记录时间、URL、响应和失败原因
      → 失败可重试，但不阻塞 CMS 发布

下架或删除时同样通知旧 URL，并保证页面返回正确的 404/410 或业务选定的重定向。当前代码管线已存在，还需 Cloudflare 配置 INDEXNOW_KEY、执行一次真实发布并验收日志。

### 5.2 CMS 发布清单

- [ ] 标题直接回答用户要找的主题。
- [ ] 首段讲清“是什么/谁可以/何时/哪里/怎么做”。
- [ ] 来源、适用范围、审核日、作者/审核人已填。
- [ ] 截止日、租金、AMI、房型等数字已复核。
- [ ] 分类、关键词、slug、canonical 和封面 alt 已填。
- [ ] 封面图不烧死短期数字，或已有下架策略。
- [ ] 结构化数据只反映可见内容。
- [ ] 发布后页面 200、列表可达、sitemap/lastmod 正确。
- [ ] IndexNow 任务可追踪，失败有日志和重试。

### 5.3 季度证据包

每季度存档：

- 固定提问集版本、产品/模型、地区、语言、登录状态和新会话设置。
- 引用 URL、完整截图、事实准确性判断。
- Search Console 生成式 AI 表现、索引和 CWV。
- Cloudflare 验证爬虫日志与挑战/阻断事件。
- GA4 AI referral 和转化，并注明数据是可观测下限。
- 本季度事实变更、文章更新、异常和修复。
