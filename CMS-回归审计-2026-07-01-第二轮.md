## 🔴 新 Bug 1（P0）：乐观锁必然触发 —— 正式后台无法编辑任何已有内容

- 【位置】`cms-demo/cms-core.mjs` `prepareEntryForSave`（`updatedAt: now`）+ `functions/_lib/content-repository.js` `requireFreshUpdatedAt` + `cms-demo/admin.js` 保存流程
- 【实测证明】模拟完整管线：库中 `updatedAt = 2026-07-01T10:00:00Z`，客户端 `prepareEntryForSave` 在发送前把 `updatedAt` 盖成当前时间再 PUT → 服务端比对 `input.updatedAt !== existing.updatedAt` → **永远不相等 → 每次保存都 409「内容已被他人修改，请刷新后再编辑」**。
- 【后果】正式环境（remote 模式）下：新建能成功（POST 无锁），但**编辑、置顶、改状态任何已有条目全部失败**。后台核心功能不可用。44 项测试没抓到它，因为测试只单测了锁本身（传入匹配的 updatedAt），没有覆盖「admin 保存 → PUT」的真实管线。
- 【修法】改用独立字段传"期望版本"：
  1. 客户端：PUT 请求体加 `expectedUpdatedAt` = 从服务端加载时的原始 `updatedAt`（在 `toRemoteEntryPayload` 里带上，取 `state.entries` 中该条目的原值，不要用 prepareEntryForSave 盖过的）。
  2. 服务端：`requireFreshUpdatedAt` 改为比对 `input.expectedUpdatedAt`（而不是 `input.updatedAt`，那个字段语义已经被"本次保存时间"占用了）。
  3. **补一条端到端测试**：模拟「加载条目 → prepareEntryForSave → PUT payload → upsertEntry(requireFreshUpdatedAt)」全链路,断言正常保存成功、并发冲突时 409。
- 【注意】POST /api/content 是 upsert 语义且无锁——带已有 id 的 POST 可绕过乐观锁静默覆盖。顺手修：POST 遇到已存在的 id 时返回 409 提示改用 PUT（或同样应用锁）。

## 🔴 新 Bug 2（P1，功能开关失效）："仅看开放中"一提交表单就失效

- 【位置】`functions/_lib/cms-core.js` 筛选表单（hidden `openOnly=0` + checkbox `openOnly=1`）+ `functions/apartments/index.js`（`url.searchParams.get("openOnly") !== "0"`）
- 【实测证明】勾选状态下提交,查询串为 `openOnly=0&openOnly=1`,`searchParams.get()` 返回**第一个值 "0"** → openOnly 判定为 false。即:默认进页开关有效,但用户一点"筛选"(即使保持勾选),开关就永久关闭,已满/过期条目混回结果。
- 【修法】服务端改 `url.searchParams.getAll("openOnly").includes("1")`（或去掉 hidden input,勾选=1、未勾选时无参数,再用一个 `filtered=1` 标记区分"用户提交过表单"与"默认进页"）。补测试:`?openOnly=0&openOnly=1` 应判定为 true。

## 🟡 小问题（顺手修）

1. **404 页面还是英文裸页**：`notFoundResponse` 仍输出 `<h1>Apartment update not found</h1>`,和新加的中文友好错误页不一致。让 404 也走 `htmlErrorResponse` 样式（保持 404 状态码,带回列表页链接）。
2. **移动端筛选面板偏高**（375px 实测约 620px:搜索+三下拉+开关+按钮）,首屏又被推远了。建议移动端用 `<details><summary>筛选</summary>…</details>` 折叠,默认收起,只露搜索框和"仅看开放中"。
3. **卡片徽章丢了房型**：`renderCardChips` 只保留 置顶/状态/地区/年龄,房型(1B/2B)对找房客户是关键信息。建议挤掉"置顶"换成第一个房型,或允许 5 个徽章。
