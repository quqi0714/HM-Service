# Huamei CMS Demo — 审查与修改清单（可直接执行）

审查日期：2026-06-30
审查对象：`cms-demo/`（华美官网新增的 CMS / 博客 / 公寓房源 演示版）
方式：静态代码审查 + 本地实跑（桌面 + 移动端截图）+ 单元测试（7 项全过）+ 语法检查（通过）

## 背景（给接手方）

华美服务中心官网是纯静态站，部署在 Cloudflare Pages（域名 huameihope.com）。
新需求：让员工登录后台发布**公寓房源更新**和**博客文章**，方便发给客户查看，并通过持续更新提升 Google 收录/推流。
目标生产架构：Cloudflare Pages + Pages Functions + Access（鉴权）+ D1（数据）+ R2（图片）+ SEO 友好路由。
当前 demo 是**前端工作流演示**：数据存浏览器 localStorage，无真实登录/数据库/服务端渲染。以下清单按严重程度排列，每条含【位置】【现象】【改法】。

## 关键文件

- `cms-demo/admin.html` / `admin.js`：后台编辑 UI
- `cms-demo/apartments.html` / `blog.html` / `detail.html` / `preview.html`：公开页

## 🔴 P0 — 上线前必须解决

### P0-1 正文富文本存储型 XSS（唯一未转义的注入点）
- 【位置】`public.js` → `renderDetail()` 里 `${entry.bodyHtml || ...}` 原样注入；富文本编辑用 `admin.js` 的 `document.execCommand`
- 【现象】其他字段（标题/摘要/标签/alt）都已正确转义（做得好），只有正文按设计保留 HTML。一旦搬到 D1，任何进入 `bodyHtml` 的脚本都会变成影响全体访客的存储型 XSS。且 `createLink` 的 URL 来自 `window.prompt` **零校验**，可注入 `javascript:`。
- 【改法】渲染前用 **DOMPurify** 按白名单净化（或保存时净化落库）；白名单标签限 `h2/h3/p/ul/ol/li/strong/em/a/br`，`a` 只允许 `http(s)/mailto` 且强制 `rel="noopener nofollow"`。

### P0-2 客户端渲染 + 查询串 URL → 对 Google 几乎无效（与 SEO 目标冲突）
- 【位置】`detail.html?type=apartment&slug=apartment-318`，全靠 JS 在浏览器渲染，`<title>`/meta 加载后才由 JS 补
- 【现象】做这功能的首要目的是提升收录，但当前形态恰恰是搜索引擎最不友好的
- 【改法】生产改为 **Pages Functions 服务端直出 HTML**：干净路径 `/apartments/apartment-318`、`/blog/<slug>`；每页真实 `<title>`、`meta description`、`canonical`、Open Graph，以及 `schema.org/Apartment`、`Article` 结构化数据；新内容自动写入 `sitemap.xml`

### P0-3 后台条件字段串显（实跑确认的功能 bug）
- 【位置】`styles.css` `.form-grid{display:grid}` 优先级压过 HTML `hidden` 属性；`admin.js` 用 `els.blogFields.hidden` 切换
- 【现象】实测：编辑「公寓」时 `blogFields.hidden=true` 但 `computedDisplay=grid`，"文章分类"仍显示；反之编辑「博客」时公寓字段也会串出。员工会对着公寓填文章分类，数据串味。
- 【改法】`styles.css` 加 `[hidden]{display:none !important;}`；或改用 class 切换（如 `.is-hidden{display:none}`）而非 `hidden` 属性

## 🟡 P1 — 上线前应修

### P1-1 公寓编号无唯一性校验 → slug 撞车
- 【位置】`cms-core.mjs` `buildApartmentSlug` = `apartment-<编号>`；无查重
- 【现象】两条记录同编号时 `findEntryBySlug` 只返回第一条，详情链接互相覆盖
- 【改法】保存时校验编号唯一并报错；D1 给 `apartment_number` 建唯一索引

### P1-2 点"预览"会偷偷把草稿写库
- 【位置】`admin.js` `openPreview()` 调 `upsertEntry()`
- 【现象】预览=保存，用户没点保存数据却被落库/改状态
- 【改法】预览走临时态/内存或一次性 token，不落库

### P1-3 徽章重复 + 状态/标签语义重叠
- 【位置】`cms-core.mjs` `APARTMENT_TAG_OPTIONS` 含"抽签中"，同时也是 `APARTMENT_STATUS_OPTIONS`；`public.js` 详情页两处都渲染
- 【现象】一条房源显示两个"抽签中"，看着像 bug
- 【改法】状态作为"可用性"唯一来源；标签只留营销词（重点推荐/全新公寓/适合长者/适合家庭）

### P1-4 存储解析失败静默覆盖用户数据
- 【位置】`cms-store.mjs` `loadEntries()` catch 里 `saveEntries(seedEntries)`
- 【现象】"出错就用种子数据覆盖"的思路不能带进 D1/R2 迁移，会造成数据丢失
- 【改法】生产迁移时对解析失败**只读不写**、告警并保留原始数据

### P1-5 详情页缺"关键信息速览表"
- 【位置】`public.js` `renderDetail()`
- 【现象】城市/租金/收入上限/申请截止/如何申请全埋在正文散文里，客户无法一眼扫到
- 【改法】详情页顶部加结构化事实面板（城市 / 房型 / 租金 / 年龄 / 状态 / 截止日 / 申请方式）+ 醒目"如何申请"CTA

### P1-6 数据模型补关键结构化字段
- 【位置】`cms-core.mjs` `createEmptyEntry` / seed / admin 表单
- 【现象】城市、租金、收入上限现在全塞在标题/摘要/正文里，无法筛选排序；`region` 只有南/北加太粗
- 【改法】新增独立字段：`city`（服务 60+ 城市，独立城市字段可做本地 SEO，如"San Gabriel 长者公寓"页吃本地流量）、`rentRange`、`incomeLimit`、`applicationDeadline`、`externalApplyLink`

## 🟢 P2 — 视觉 / 交互 / 产品化打磨

### 视觉 & 移动端
- **P2-1 移动端 hero 瘦身**：公寓列表页手机上要滚过大标题+三行说明+筛选栏才看到第 2 张卡片。压成一行小标题，筛选栏 sticky 吸顶，让房源尽快进入视野。
- **P2-2 卡片密度**：左缩略图方向对，但几百套时仍偏高，再紧凑一档或加"列表/大图"切换。
- **P2-3 复用主站视觉**：demo 自建了 `styles.css` + Google Fonts，header/footer 也是重写的。生产应共用主站导航/页脚/字体，避免像"另一个网站"。
- **P2-4 硬编码指标**：`apartments.html` hero 里"2 公开示例""3 筛选维度"写死，不随真实数据变，生产移除。

### 功能完整度
- **P2-5 列表分页 / 懒加载**：几百套一次全渲染会卡、首屏慢，必须分页或无限滚动。
- **P2-6 排序**：加"最新更新 / 租金高低"排序。
- **P2-7 "仅看开放中"快捷开关**：已满/过期保留页面利于 SEO，但客户浏览时想一键只看可申请的。
- **P2-8 过期状态自动化**：按 `applicationDeadline` 自动置灰"已过期"，别靠人工改。
- **P2-9 分享给客户**（原始痛点）：每套房源可复制单独链接、生成微信分享卡（OG 图/标题/摘要），进阶可生成"某客户专属房源清单"。
- **P2-10 图片处理**：员工手机原图 3-5MB，上线需自动压缩 + 转 WebP + 生成缩略图（R2 / Cloudflare Images）。
- **P2-11 移动端后台可用性**：员工可能在手机上更新房源，后台当前偏桌面，需专门验证移动编辑体验。

### 代码质量
- **P2-12 富文本编辑器**：`document.execCommand` 已废弃且行为不一致，生产换 Tiptap/Quill 或至少保证输出净化。
- **P2-13 测试覆盖**：现有 7 项只覆盖 slug/筛选/日期；补渲染层、净化、URL 生成的测试。
