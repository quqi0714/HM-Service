## 🔴 P0：部署阻断项（按顺序修）

### 工单 A：sitemap 回归 —— 动态 sitemap 把现有主站页面全部弄丢

- 【文件】`functions/sitemap.xml.js`、`functions/_lib/cms-core.js` 的 `buildSitemapXml`
- 【问题】`_routes.json` include 了 `/sitemap.xml`，函数会覆盖根目录静态 sitemap.xml。实测函数输出**只含 CMS 条目**：主站已收录的 `/`、`/vehicle.html`、`/health.html`、`/love-health.html` 全部消失，且 `/apartments`、`/blog` 两个列表页也不在输出里。上线即损害现有 SEO。
- 【要求】`buildSitemapXml` 合并输出三类 URL：
  1. 静态页（可硬编码）：`/`（priority 1.0）、`/vehicle.html`、`/health.html`、`/love-health.html`
  2. 列表页：`/apartments`、`/blog`
  3. 全部已发布 CMS 条目（现有逻辑）
- 【验收】本地调用 `buildSitemapXml`，输出同时含 `https://huameihope.com/`、`/vehicle.html`、`/apartments` 和 `/apartments/318` 类条目；给该函数补一条单元测试断言静态页存在。

### 工单 B：生产版 /apartments 列表页缺筛选、搜索、分页（立项核心需求）

- 【文件】`functions/_lib/cms-core.js` 的 `renderListPage`、`functions/apartments/index.js`、`functions/_lib/content-repository.js` 的 `listEntries`
- 【问题】筛选功能只存在于本地演示（cms-demo/apartments.html 的 localStorage 版）。客户实际访问的生产 SSR 页是纯静态列表：无搜索、无地区/年龄/房型筛选、无分页，且 `limit: 200` 硬截断。几百套房源时既筛不了也看不全。
- 【要求】
  1. SSR 列表页加筛选控件：搜索框 + 地区（南加/北加）+ 年龄（18+/55+/62+）+ 房型（1B/2B/3B/3B+）下拉 + **"仅看开放中"开关**（默认开，隐藏已满/已截止/已过期）。
  2. 实现方式二选一（推荐 1）：
     - 方式 1：服务端把已发布条目数据以 JSON 内嵌进页面（`<script type="application/json">`，注意用 `<` 转义防打断），轻量客户端 JS 复用 demo 的 `filterApartmentEntries` 逻辑做即时筛选。首屏 HTML 仍服务端渲染全量卡片，保证 SEO。
     - 方式 2：`?region=&age=&room=&page=` 查询参数走服务端筛选渲染（listEntries 增加对应 WHERE），每次筛选整页刷新。
  3. 分页：服务端 `?page=N`（每页 24-50 条），页脚输出上一页/下一页链接（真实 `<a href>`，SEO 可爬）；`listEntries` 支持超过 200 条的场景。
- 【验收】模拟 250 条已发布公寓：列表页可翻页看到全部；筛选"北加州 + 62+ + 1B"结果正确；"仅看开放中"隐藏已截止条目；禁用 JS 时首屏仍能看到第一页完整卡片 HTML。

### 工单 C：移动端列表卡片徽章竖排堆叠，卡片过高（375px 实测复现）

- 【文件】`functions/_lib/cms-core.js` 的 `renderBaseCss()`（`.entry-card__thread .entry-meta{justify-content:flex-end}`）与 `renderEntryCard` / `renderChips`
- 【问题】375px 宽实测：卡片头部日期居左、徽章右对齐，6+ 个徽章（置顶/南加州/62+/1B/2B/抽签中/重点推荐）在右侧**竖着摞成一列**，一张卡片占大半屏。客户主要用手机浏览，密度要求完全落空。（800px 以上正常。）
- 【要求】
  1. ≤680px 断点下：徽章不再右对齐挤在窄列，改为整行自然换行（`justify-content:flex-start; width:100%`，date 与 chips 分两行）。
  2. `renderEntryCard` 卡片上限输出 3-4 个徽章，优先级：申请状态 > 地区 > 年龄要求，其余只在详情页展示（详情页 `renderChips` 保持全量）。
- 【验收】375px 视口下单卡高度明显收敛（目标 ≤ 屏高 40%），徽章横向排列不竖摞。

### 工单 D：鉴权配置加固 + 部署顺序

- 【文件】`functions/_lib/http.js`、`wrangler.toml`
- 【问题与要求】
  1. **D1 - `CMS_AUTH_BYPASS` 保险**（http.js:67）：`env.CMS_AUTH_BYPASS === "true"` 直接放行全部写入。加双保险：bypass 生效同时要求 `new URL(request.url).hostname` 属于 `localhost/127.0.0.1/::1`，否则忽略 bypass。代码处加中文注释警告"生产环境禁止设置此变量"。
  2. **D2 - `CMS_ADMIN_EMAILS` 空值 = 放行所有 Access 用户**：当前 allowlist 为空时跳过邮箱校验（wrangler.toml 默认就是 `""`）。改为 **fail-closed**：远程模式下 allowlist 为空时抛 503 "CMS_ADMIN_EMAILS 未配置"，强迫上线前必须填。
  3. **D3 - JWT 增强**：`verifyCloudflareAccessJwt` 补 `iss` 校验（应等于 `https://<team-domain>`）；`fetchAccessJwk` 加模块级缓存（Map + 1 小时过期），避免每次 API 调用都拉 Access certs。
  4. **D4 - 部署顺序写进文档**：首页 3 个入口已指向 `/apartments`，若先发首页后配 D1，客户点入看到 JSON 500。正确顺序：预览环境配好 D1/R2/Access 并验证 → 再发正式域名。
- 【验收】单测覆盖：bypass 在非 localhost 主机名下无效；allowlist 为空时远程请求得到 503；iss 不匹配得到 401。

### 工单 E：cms-demo 目录上线处置

- 【文件】`_routes.json`、`robots.txt`、部署配置
- 【问题】demo 列表页带 localStorage 假种子数据，部署到正式域名会呈现假房源并与 `/apartments` 形成重复内容；admin.html 是公开静态文件（API 有 JWT 保护，但后台界面全网可见）。robots.txt 目前完全没提 cms-demo。
- 【要求】三选一并落实：(a) 生产构建时排除 `cms-demo/`（推荐，本地保留演示能力）；(b) 改名为 `/admin/` 并用 Cloudflare Access 保护整个路径；(c) 至少 robots.txt 加 `Disallow: /cms-demo/` 且 Access 覆盖该路径。
- 【补充】`index.html` 入口切换脚本的本地主机判定（`['localhost','127.0.0.1','::1']`）与 `cms-backend.mjs` 的 `LOCAL_HOSTS`（多含空字符串，即 file://）不一致——统一都把 `''` 算作本地。
