## 二、🔴 P0 — 部署阻断项（上线前必须解决）

### P0-A sitemap 回归：一上线，现有主站从 Google 索引地图里消失

- 【位置】`functions/sitemap.xml.js` + `_routes.json`（include 含 `/sitemap.xml`）
- 【现象】动态 sitemap 函数会**覆盖**根目录静态 `sitemap.xml`。实测函数输出**只含 CMS 条目**——首页 `/`、`vehicle.html`、`health.html`、`love-health.html` 全部消失，连 `/apartments`、`/blog` 两个列表页本身也不在里面。
- 【后果】这与"提升 Google 推流"的目标直接相反：已上线且有排名的 4 个页面从 sitemap 消失。
- 【改法】`buildSitemapXml` 合并输出：静态页（/、/vehicle.html、/health.html、/love-health.html）+ 两个列表页（/apartments、/blog）+ 全部已发布 CMS 条目。静态部分可硬编码在函数里。

### P0-B 正式 `/apartments` 列表页没有筛选、搜索、分页——核心需求缺失

- 【位置】`functions/_lib/cms-core.js` `renderListPage` / `functions/apartments/index.js`
- 【现象】demo 版（cms-demo/apartments.html）有搜索+地区+年龄+房型筛选，但那只在本地演示存在；**客户实际访问的生产页是纯静态列表，什么筛选都没有**，且 `limit: 200` 硬截断——几百套房源时既筛不了也看不全。
- 【后果】"客户按南北加州/年龄/房型快速筛选几百套房源"是这个系统的立项需求，生产页面没兑现。
- 【改法】给 SSR 列表页加一段轻量客户端 JS（复用 demo 的筛选逻辑，数据从服务端注入或读一个公开只读 JSON 接口）+ 分页（`?page=2` 服务端分页，SEO 友好）。至少上线时要有：搜索框 + 三个下拉 + "仅看开放中"开关。

### P0-C 移动端列表卡片徽章竖排堆叠，卡片高得离谱（375px 实测）

- 【位置】`renderBaseCss()` 中 `.entry-card__thread .entry-meta{justify-content:flex-end}`
- 【现象】375px 宽度实测：日期左侧、徽章右侧对齐，6 个徽章（置顶/南加州/62+/1B/2B/抽签中/重点推荐）**在右侧竖着摞成一列**，一张卡片占大半屏。800px 以上显示正常。
- 【后果】与"移动端高密度快速浏览几百套"的需求相反；客户大多从小红书/微信用手机打开。
- 【改法】窄屏下徽章换行铺满整行而非右对齐竖排；卡片上限显示 3-4 个徽章（状态+地区+年龄优先），其余进详情页。

### P0-D 鉴权配置三连雷 + 部署顺序风险

1. **`CMS_AUTH_BYPASS === "true"` 直接放行所有写入**（`functions/_lib/http.js:67`）。这是给本地开发用的，但如果误在 Cloudflare 环境变量里设了它,整个后台 API 对全网裸奔。→ 建议加双保险：bypass 时同时校验 `request.url` 主机为 localhost，或干脆只在 `wrangler pages dev` 的 `.dev.vars` 里存在、代码里加注释警告。
2. **`CMS_ADMIN_EMAILS = ""`（wrangler.toml 默认值）= 不限制邮箱**：`requireAdmin` 里 allowlist 为空时跳过校验，任何通过 Access 认证的人都能写。→ 上线前必须填入真实邮箱列表，且 Access 策略本身也要限定邮箱。
3. **后台静态页在 `_routes.json` 的 exclude 里**（`/cms-demo/*`），即 admin.html 是公开静态文件。API 有 JWT 保护（好，纵深防御成立），但 Access 策略必须同时覆盖后台页面路径，否则后台界面全网可见（信息暴露 + 钓鱼面）。
4. **部署顺序**：首页 3 个入口已指向 `/apartments`。若先部署代码、后配 D1，客户点进去会看到 `{"error":"Internal server error"}` 的 JSON。→ 正确顺序：**先在 Cloudflare 预览环境配好 D1/R2/Access 并验证，再把带入口的首页发到正式域名**。

## 三、🟠 cms-demo 目录的上线处置（handoff 里自己也问了）

- **建议：`cms-demo/` 不随生产部署**（或至少 robots.txt 加 `Disallow: /cms-demo/`——当前 robots.txt 没提它）。理由：demo 列表页带 localStorage 种子数据，会在正式域名上呈现假房源 + 和 `/apartments` 形成重复内容；admin 页暴露后台结构。
- 若保留后台在线上，建议改名为 `/admin/` 并由 Access 整路径保护。后台自动切换 remote 模式的机制（`cms-backend.mjs` 按 hostname 判断）设计是好的，改名不影响。
- 小不一致：`cms-backend.mjs` 的 LOCAL_HOSTS 含空字符串（file:// 算本地），但 `index.html` 里的入口切换脚本不含——`file://` 直接打开首页时"公寓清单"仍指向 `/apartments`（断链）。统一为都含 `''`。

## 四、🟡 P1 — 上线前应修

1. **详情页/列表页没有页脚、没有"咨询梅老师"联系入口**（实测滚到底：外部申请链接按钮之后页面直接结束）。客户看完房源的下一步动作应该是找华美，而现在唯一 CTA 是把人送去项目方官网。→ 详情页底部加联系区块（电话/微信/回首页咨询），列表页加页脚。这是转化断点，强烈建议上线前补。
2. **SSR 页面视觉与主站脱节**：极简自带 CSS（Georgia serif、无 logo、无主站导航/页脚）。客户从首页点"公寓清单"过去像进了另一个网站。→ 至少统一：logo + 主导航（首页/住房/车辆/健康）+ 页脚 + 字体（Marcellus/Noto Serif SC）。
3. **事实面板缺"城市"行**：`renderApartmentFacts` 有地区（南/北加）但没有 city 字段行——城市恰恰是客户最关心的（也是本地 SEO 词）。JSON-LD 里有,页面上没有。
4. **HTML 路由出错时返回 JSON**：`/apartments` 在 DB 缺失/异常时 `handleError` 返回 JSON 500,访客看到裸 JSON。→ HTML 路由包一层友好错误页（"内容加载中，请稍后再试"+ 联系方式）。
5. **并发编辑最后写赢**：PUT 全字段覆盖,两个员工同时编辑同一条会互相覆盖且无提示。→ 简单方案：带 `updatedAt` 做乐观锁,不匹配返回 409"内容已被他人修改"。
6. **JWKS 每请求都拉取**（`fetchAccessJwk` 无缓存）：每次 API 调用都请求 Access certs,慢且可能被限流。→ 模块级缓存 1 小时。顺带补 `iss` 校验（== `https://<team>.cloudflareaccess.com`）。
7. **列表卡片徽章数量失控**：`renderChips` 在卡片上输出全部（置顶+地区+年龄+全部房型+状态+全部标签）。→ 卡片限 3-4 个,详情页显示全部。

## 五、🟢 P2 — 功能/UX 打磨（上线后迭代亦可）

1. **排序控件**：最新更新 / 截止日临近 / 租金。
2. **"仅看开放中"**默认开关（已满/过期页面保留给 SEO,浏览时默认隐藏）。
3. **过期自动化**：按 `applicationDeadline` 自动把状态渲染为"已截止"（不必改库,渲染时判断即可）。
4. **图片压缩管线仍缺失**：上传原样存 R2（8MB 上限）。员工传手机原图,客户手机端加载 5MB 封面。→ 接 Cloudflare Images 或上传前浏览器端压缩（canvas 转 WebP,最长边 1600px）。
5. **上传安全小补**：服务端只信 `file.type`（客户端可伪造）。风险低（存储 content-type 固定、key 随机、不允许 svg）,但可加 magic bytes 嗅探。
6. **归档语义**：DELETE 实际是归档（数据保留）——设计合理,但后台按钮叫"删除"会误导员工以为可彻底移除。按钮文案改"下架/归档",并考虑提供真删除给管理员。
7. **`archiveEntry` 对不存在的 id 也返回 ok**：应先查再改,返回 404。
8. **OG 分享卡缺尺寸/站点名**：`og:site_name`、`og:image:width/height`、`twitter:card` 未输出,微信/Twitter 卡片兼容性打折。
9. **列表页缓存**：`max-age=120` 合理;但发布新房源后想立即可见,可在后台保存成功提示里注明"公开页 2 分钟内更新"。
10. **后台移动端可用性**仍未验证（员工可能手机发房源）。
11. **审计日志只记 id/action/email**,不记变更内容摘要——排查"谁把租金改错了"时无从查起。可存 title + 状态快照。

## 六、建议的部署顺序（checklist）

1. 修 P0-A sitemap 合并、P0-B 列表筛选+分页、P0-C 移动端卡片、P1-1 联系 CTA/页脚。
2. Cloudflare 后台操作（handoff 已列,补充顺序）：
   - 创建 D1 → 填 `database_id` → 跑 migration
   - 创建 R2 bucket `huamei-cms-assets`
   - 配 Access：策略覆盖 **后台页面路径 + `/api/*`**,限定邮箱
   - 环境变量：`CF_ACCESS_TEAM_DOMAIN`、`CF_ACCESS_AUD`、`CMS_ADMIN_EMAILS`（必填！）;确认 **没有** `CMS_AUTH_BYPASS`
3. 在 **预览环境**（`*.pages.dev`）完整走一遍：登录 → 发布公寓 → 传图 → 置顶 → 下架 → 打开 `/apartments`、`/apartments/318`、`/sitemap.xml` 逐一核对。
4. 决定 cms-demo 处置（不部署 / 改名 /admin + Access + robots Disallow）。
5. 正式发布首页 + 系统;Google Search Console 重新提交 sitemap。
6. 上线后 24 小时内检查:GSC 抓取状态、GA 是否统计到新页面（注意:SSR 页面目前**没有埋 GA 代码**——统一模板时记得加上,否则新栏目流量不可见）。
